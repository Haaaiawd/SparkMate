# ADR-001：技术栈选择

**状态**: Accepted
**日期**: 2026-03-09
**决策者**: 研究团队

---

## 背景

项目「星伴 (SparkMate)」是一款以微信小程序为载体的大学生 AI 伴学系统，需要在以下约束下选择技术栈：

**硬约束**:
- 目标平台：微信小程序（用户端），Web（研究管理后台）
- 团队规模：4-5 人，以 AI 辅助编程为主
- 开发周期：约 1 学期内完成 MVP
- 预算：学生 SRTP 项目，低成本优先，优先使用国内云服务
- 数据合规：涉及用户情绪/心理数据，需要国内合规存储

**软约束**:
- 接入 LLM API（主要用于对话干预）
- 需要 Cron Job 能力（每日数据探针和量表推送）
- 研究后台需要 Web 界面 + CSV 导出

---

## 候选方案对比

### 前端（微信小程序）

| 候选 | 总分 | 优势 | 劣势 |
|------|:----:|------|------|
| **原生微信小程序 (WXML/WXSS/JS)** | 44/55 | 文档最全、微信 API 兼容最佳、无框架依赖 | 语法独特，组件化较弱 |
| **uni-app (Vue3)** | 41/55 | 跨平台潜力、Vue 生态熟悉、组件化好 | 编译层偶有兼容坑，调试略繁琐 |
| **Taro (React)** | 38/55 | React 生态、TypeScript 友好 | 与微信 API 的适配层问题较多 |

**决策**: 选用 **原生微信小程序** + TypeScript（小程序基础库 3.x 已支持 TS）

**理由**:
- Lottie 动画、`wx.setKeepScreenOn`、`wx.vibrateShort`（戳一戳）等关键 API 原生支持最稳定
- 微信订阅消息推送（晚点名提醒、量表推送）在原生环境下调试最便捷
- 团队规模小，减少跨框架转换成本是更重要的目标

---

### 后端 + 数据库

| 候选 | 总分 | 优势 | 劣势 |
|------|:----:|------|------|
| **微信云开发 (CloudBase)** | 50/55 | 无服务器、内置数据库+云函数+推送、与小程序深度集成、免运维 | 数据查询能力略弱于关系型数据库、学习曲线存在 |
| **Node.js + Express + MySQL (腾讯云)** | 42/55 | 灵活、关系型数据强 | 需自行搭建服务器、运维成本高 |
| **Python FastAPI + PostgreSQL (腾讯云)** | 40/55 | 强大、AI/数据处理好 | Python 生态对小程序集成不如 Node 原生 |

**决策**: 选用 **微信云开发（腾讯云 CloudBase）**

**理由**:
- 云函数（Cloud Functions）覆盖所有后端逻辑：业务 API、Cron Job（定时触发器）、LLM 调用
- 云数据库（MongoDB 兼容）满足数据存储需求；研究数据导出通过 CloudBase 控制台或 Admin SDK
- 云托管（Cloud Run）可部署研究管理后台（Next.js）
- 订阅消息推送直接通过微信 API + 云函数实现，无需额外消息队列
- **对 4-5 人学生团队最关键**：零服务器运维，按量计费，学生账号有免费额度

---

### LLM API

| 候选 | 总分 | 优势 | 劣势 |
|------|:----:|------|------|
| **DeepSeek API (DeepSeek-V3)** | 50/55 | 国内合规、价格极低（约 ¥0.27/M tokens）、响应速度快 | 知名度不如 GPT，但技术上已达 GPT-4 级别 |
| **通义千问 API (Qwen-Plus)** | 44/55 | 阿里云生态、国内合规 | 与云开发集成成本略高 |
| **文心一言 API (ERNIE)** | 38/55 | 百度生态 | 接入复杂，文档体验差 |
| **OpenAI GPT-4o** | 30/55 | 能力最强 | 国内访问受限，需要代理，不合规 |

**决策**: ⚠️ **待人类确认（PENDING）**

候选方案倾向：DeepSeek API（deepseek-chat），但需要团队确认以下事项后最终敲定：
- [ ] 是否已有其他 LLM 服务合作或偏好？
- [ ] SRTP 经费是否覆盖 API 成本？
- [ ] 学校/导师是否有数据合规方面的特殊要求？

**候选方案参考**:
- DeepSeek API：国内合规、价格极低（约 ¥0.27/M tokens）、技术质量高
- 通义千问 API（Qwen-Plus）：阿里云生态、国内合规
- 文心一言 API（ERNIE）：百度生态，接入较复杂

---

### 研究管理后台

| 候选 | 总分 | 优势 | 劣势 |
|------|:----:|------|------|
| **Next.js（部署到 CloudBase 云托管）** | 48/55 | React 生态、SSR/SSG、部署简单 | 对于简单后台略显重 |
| **CloudBase 内置 Web 控制台** | 40/55 | 零开发成本 | 定制化能力不足，无法做专项仪表盘 |
| **Vue3 + Vite（部署到 CloudBase）** | 45/55 | 轻量、快速 | 生态略弱于 Next.js |

**决策**: 选用 **Next.js App Router + TailwindCSS + shadcn/ui**，部署到微信云托管

**理由**:
- 研究后台核心需求是数据展示（图表）+ CSV 导出，Next.js 生态（Recharts 等）完全满足
- 与 CloudBase Admin SDK 集成简单（服务端直接访问云数据库，无需额外 API 层）
- shadcn/ui 组件库提供开箱即用的 Table、Chart，减少 UI 开发时间

## 最终技术栈总览

| 层级 | 技术选型 | 版本/说明 |
|------|---------|---------|
| **小程序前端** | 微信原生小程序 + TypeScript | 基础库 3.x |
| **动画** | Lottie-miniprogram | 微光崽动画 |
| **后端/云函数** | 微信云开发（CloudBase）Node.js 18 云函数 | 无服务器架构 |
| **数据库** | CloudBase 云数据库（MongoDB 兼容）| 文档型数据库 |
| **LLM** | DeepSeek API (deepseek-chat) | HTTP API 调用 |
| **推送** | 微信订阅消息 API | 晚点名/量表推送 |
| **定时任务** | CloudBase 定时触发云函数 | Cron 表达式 |
| **研究后台** | Next.js 15 + TailwindCSS + shadcn/ui | 部署到云托管 |
| **图表库** | Recharts | 数据可视化 |

## 权衡点

1. **CloudBase vs 自建服务器**: 牺牲部分灵活性（无纯 SQL 查询），换取零运维和快速开发
2. **MongoDB vs PostgreSQL**: 文档型 DB 对嵌套数据（如 items[]）更自然，但联表查询能力弱；研究数据导出为 CSV 后用 SPSS 处理，不依赖后端复杂查询
3. **DeepSeek vs 本地模型**: 放弃零 API 成本，换取稳定服务和良好对话质量
4. **Next.js 重 vs 纯静态页面轻**: 选择 Next.js 是为了 Server Action（安全访问 CloudBase Admin SDK）

## 后果

**正面**:
- 开发速度快，4-5 人团队 1 学期内完成 MVP 可行
- 运维成本极低（CloudBase 免费额度覆盖 MVP 阶段）
- LLM 月费估算 < ¥50，不超出 SRTP 预算

**负面**:
- CloudBase 数据库学习曲线（与 SQL 思维不同）
- 未来如需迁移到独立服务器，需要重构云函数层

**后续行动**:
- 开发前申请 SRTP 经费覆盖 CloudBase 和 DeepSeek API 费用
- 在 Onboarding 知情同意书中明确数据存储在腾讯云的说明
